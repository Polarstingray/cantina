Building Cantina

A step-by-step field guide to how a small household grocery & meal tracker was built — from a single Python file to a secure, multi-user web app — and the potholes hit along the way.

Python 3.12 FastAPI SQLite Vanilla JS (no build step) 21 commits Novice-friendly

Cantina is a web app a family runs on the home network to keep track of the food they own. You catalog foods (with cost and nutrition), compose them into meals (recipes), track what's on hand in your inventory, see which meals you can build right now (the menu), keep a shared grocery list, and log your spending.

That's the finished product. But the interesting part — and the reason this guide exists — is how it got there. The project grew in clear stages, each recorded as a git commit, and each stage taught a lesson the hard way. We're going to walk that whole road in order, replicate the key code, and stop at every pothole to understand what went wrong and how it was fixed. By the end you should be able to build something similar yourself.

The stack, and a map of the finished app

Three layers talk to each other. A web browser runs the frontend (plain HTML/CSS/JavaScript). It sends requests to the backend (Python, using a framework called FastAPI). The backend reads and writes a database (SQLite, a single file on disk).

Prerequisites & setup

You need Python 3.12 and a terminal. Everything in the backend uses Python; the frontend is plain files served by the backend, so there's no Node.js, no webpack, no build step. That is a deliberate choice we'll come back to.

A virtual environment is a private folder of Python packages for one project, so installing FastAPI here doesn't pollute your whole computer (or clash with another project). You create one, "activate" it, and install into it.

Chapter 1
The idea & the domain model

commits: 9d9e013ad22293273fb4c

Goal: before any database, any web server, any login — describe the things the app is about. For a grocery tracker, that's two ideas: a Food and a Meal. Everything else is built on top of these.

💡 Concept — a class

A class is a blueprint for a kind of thing. Food is a blueprint that says "every food has a name, a cost, some macros…". An instance is one real thing made from the blueprint, like an apple. Think recipe card (class) versus the actual cake you bake from it (instance).

The first commit created foods.py with the Food class. A food carries a name, where to buy it, a cost, nutrition ("macros"), and a description:

The list → dictionary pivot

A Meal is a bundle of foods. The very first version stored its ingredients as a plain list: [apple, apple, apple, crust]. That works, but it can't express "0.5 cups of flour" cleanly, and counting "how many apples?" means scanning the whole list. The second commit changed the ingredients into a dictionary that maps each food to an amount.

✅ Why it matters

The {food: amount} shape is the single most important early decision. It supports fractional amounts (half an egg), instant lookup of one ingredient, and totals that are a simple loop. Every later layer — recipes, "what can I build", the database schema — inherits this shape.

Giving objects an identity

The third commit added two special methods, __hash__ and __eq__, keyed on the food's name:

💡 Concept — why __hash__/__eq__?

To use an object as a dictionary key (which is exactly what {food: amount} does), Python needs to know when two objects are "the same" (__eq__) and how to file them in a lookup table (__hash__). By keying both on name, two Food objects with the same name are treated as one. That's how the catalog enforces "one entry per name" later.

🔧 Try it

python backend/src/foods.py — the file has a main() that builds an apple pie and prints its macros and price. No web server needed; it's just objects.

Chapter 2
First persistence: pickle → JSON → .bin

commits: e1aa9daa8549b326584709959fbf

Goal: objects vanish when the program exits. To remember foods between runs we need persistence — saving to disk and loading back. This chapter is a great example of trying the obvious thing, learning why it's wrong, and improving in two steps.

💡 Concept — serialization

A Python object lives in memory as a web of references. Serialization means flattening it into a stream of bytes you can write to a file; deserialization is rebuilding the object from those bytes. The question of which format to flatten into is the whole story of this chapter.

Attempt 1: pickle (and why it was dropped)

Python ships a built-in serializer called pickle. It's a one-liner, so it was the first thing tried:

⚠️ Pitfall — pickle is opaque and unsafe

A .pkl file is unreadable binary — you can't open it to inspect or fix your data. Worse, loading a pickle can execute arbitrary code, so it's dangerous to load any file you didn't write yourself. And it locks the data into Python's internal format, making a future move to a real database painful. The fix: serialize to JSON instead.

Attempt 2: JSON via factory methods

The next commits taught each class to convert itself to and from a plain dictionary. The pattern is a to_json() method (object → dict) and a create() factory (dict → object). Crucially, every dict carries a "type" field so we know whether to rebuild a Food or a Meal:

✅ Why it matters — the defensive create()

Notice create() returns None for malformed input and pads missing macros instead of crashing. Real saved data drifts over time (you add a field later; old rows lack it). A forgiving deserializer means an old or hand-edited record loads with sensible defaults rather than taking down the whole app.

Attempt 3: JSON encoded into a .bin file

The final step wrote the list of dicts as JSON text, encoded to UTF-8 bytes, into a file named data.bin:

The commit message itself spells out the intent: "this will allow for seamless migration into real databases." That foresight pays off in Chapter 7 — because the rest of the app only ever calls read_json_from_bin / write_json_to_bin, those two functions become the single seam where we later swap in SQLite without touching anything else.

💡 Concept — a "seam"

A seam is a narrow boundary in your code where you can change what's behind it without disturbing what's in front. By funneling all reads/writes through two functions, the project created a seam between "business logic" and "storage." Good seams are what make big changes (like switching databases) safe later.

Chapter 3
Inventory, Menu & the FastAPI scaffold

commits: d1adc772b71093

Goal: add two pieces of real logic — tracking what you have (inventory) and computing what you can cook (menu) — then expose everything over the web so a browser can use it.

Inventory: what's on hand

The catalog (Chapter 2) is the library of all foods that exist. Inventory is a separate, smaller thing: how many of each you currently own. It's stored as two name→quantity maps, one for raw foods and one for prepared meals (leftovers):

💡 Concept — the 1e-9 tolerance & the 0/-1 convention

Computers store decimals imperfectly: 1.0 - 0.7 - 0.3 can equal -0.0000000000000002 instead of 0. The tiny 1e-9 ("one billionth") fudge factor stops that floating-point dust from leaving phantom "0.0000001 eggs" in your inventory. Separately, these functions return 0 for success and -1 for failure — a simple convention the web layer translates into HTTP status codes.

Menu: what can I build right now?

The menu doesn't store anything — it computes. For each meal, how many full copies can you make from the foods on hand? That's the minimum, across all ingredients, of "how much you have ÷ how much the recipe needs," rounded down:

If a pie needs 3 apples and 1 crust, and you have 7 apples and 2 crusts, you can build min(floor(7/3), floor(2/1)) = min(2, 2) = 2 pies. This is where the Chapter 1 {food: amount} decision quietly earns its keep.

The FastAPI scaffold

So far everything runs only in the terminal. To let a browser use it, we put an HTTP API in front of the logic.

💡 Concept — web API, endpoints, HTTP verbs

A browser talks to a server by sending requests to URLs. An endpoint is one URL the server knows how to answer, like /foods. The verb says what you want: GET = "give me data," POST = "create something," PUT = "replace something," DELETE = "remove something." The data going back and forth is usually JSON — the same dict-and-list format we already serialize to. FastAPI is the Python library that turns a plain function into an endpoint.

💡 Concept — Pydantic input validation

That food: FoodIn annotation is doing a lot. FoodIn is a Pydantic model — a schema describing what a valid request body looks like. FastAPI automatically rejects (with a clear error) any request whose data doesn't fit: missing name, a negative cost, a name that's too long. You write the shape once; bad input never reaches your logic.

✅ Why it matters — validate at the edge

Blocking / and null bytes in names isn't paranoia: names get used in URLs and could be abused for path tricks. Rejecting garbage at the boundary means every function deeper in keeps a simple promise — "the name is clean" — and never has to re-check.

Chapter 4
The frontend SPA

commits: a81d5dda3ef32ac695e9f

Goal: a real web interface — and, just as importantly, make the backend safe to leave running on the home network.

💡 Concept — a Single-Page App (SPA)

Traditionally each click loads a brand-new HTML page from the server. An SPA loads one page once, then JavaScript redraws parts of it as you navigate, fetching only data (JSON) in the background. Cantina's SPA is deliberately tiny: plain JavaScript, no framework, no build step — just three files the browser loads directly.

The backend serves the frontend (one origin, no CORS)

Rather than running the frontend on a separate web server, FastAPI serves the static files itself. The API routes are registered first; then the frontend folder is "mounted" at / so anything not matched by an API route falls back to the page:

✅ Why it matters — same origin kills a whole class of pain

When the page and the API share one address (same origin), the browser sends cookies automatically and there's no CORS (Cross-Origin Resource Sharing) configuration to fight with. CORS is the browser's rule that a page from site A can't freely call site B. By serving both from one place, that rule simply never applies — simpler and a smaller attack surface.

api.js — the map of the backend

The frontend's api.js has exactly one function per endpoint. Read it top to bottom and you've seen the whole backend surface. A single tiny wrapper does every request:

State + re-render, and hash routing

The app keeps one state object in memory. After any change it re-fetches everything in parallel and repaints. Navigation uses the part of the URL after # (the "hash"), which the browser never sends to the server — so different "pages" like #/inventory need no extra backend routes.

Pitfall: torn files under concurrent access

Once a browser (and maybe several family phones) can hit the backend at once, two requests might try to write data.bin at the same moment. With the naïve "open, write" from Chapter 2, one write can land in the middle of another, leaving a half-written, corrupt file. The fix shipped in this chapter (commit a81d5dd) and is worth studying:

⚠️ Pitfall — never overwrite a file in place

The trick is step 5: os.replace is atomic — the operating system guarantees readers see either the entire old file or the entire new one, never a half. So you write a complete temp file first, sync it to disk, then swap it in as one indivisible step. A threading.Lock serializes writers, and three rotating .bak copies give a manual undo. (Note the comment in the real code: this is safe for one server process only — which becomes the motivation for SQLite in Chapter 7.)

Pitfall: 1.0 vs 1

Commit c695e9f let amounts be fractional everywhere (the UI steps by 0.5). But now the screen showed ugly values like "1.0 eggs" or floating-point dust like "0.30000000000000004." A tiny formatter fixes the display:

✅ Why it matters

The 1e-9 trick on the backend (Ch.3) and fmtQty on the frontend are two halves of the same lesson: floating-point numbers need cleanup both when you store them and when you show them. Forget either and users see weirdness.

Barcodes & money

commits: 184609b0fb4111275622b3070c9048ce39b

Goal: two "real product" features. Scan a barcode to auto-fill a food's details, and track how much money you spend.

Barcode lookup via OpenFoodFacts

lookup.py calls a free public database (OpenFoodFacts) and reshapes the answer into the same format the add-food form expects. It uses only Python's standard library — no new dependency:

💡 Concept — caching & graceful degradation

The _CACHE dict remembers recent lookups so scanning the same item twice doesn't hit the network twice. And every failure path returns None rather than raising — if OpenFoodFacts is down, you simply type the food in by hand. Optional features should degrade gracefully, never break the core app.

⚠️ Pitfall — the camera scanner isn't universal

Commit 0fb4111 discovered the browser's BarcodeDetector API only exists on some browsers (Chrome/Edge on Android) and needs HTTPS to use the camera at all. iOS Safari has neither. The fix: feature-detect and hide the camera button where it won't work, while the manual "type the barcode" path always remains. Always assume a browser feature might be missing.

⚠️ Pitfall — silent form failures

The same commit fixed a confusing bug: the browser's built-in form validation would silently block a submit (e.g. a stray step mismatch on the sodium field) with no message. The fix turned off native validation (form.noValidate = true) and validates in JavaScript so the specific failing field is shown in a status banner. Invisible errors are worse than loud ones.

Chapter 6
Phase 0 — the safety net

commit:a4aee07

Goal: the app works, but a big rewrite is coming (SQLite). Before touching the foundation, make the project reproducible and testable so we can prove the rewrite changed nothing the user can see.

✅ Why it matters — the order of operations

This is the single most mature decision in the whole history: write the safety net before the risky change, not after. Tests written now capture exactly how the app behaves today. After the SQLite migration, if the same tests still pass, the migration is provably invisible to users.

config.py — settings from the environment

Until now, paths and ports were hardcoded. config.py reads them once from environment variables, with sensible defaults, so the data can live outside the code folder (and survive git operations):

💡 Concept — environment variables

An environment variable is a setting you hand a program when you launch it, from outside the code (e.g. CANTINA_PORT=8080 python api.py). It keeps machine-specific choices (where files live, which port) out of the source, so the same code runs on a laptop and a server unchanged.

Characterization tests

💡 Concept — a characterization test

A regular test checks "does this do what it should?" A characterization test checks "does this still do what it currently does?" — you pin down today's behavior so a refactor can't change it without you noticing. Perfect before a migration.

The cleverest piece is the test setup (conftest.py). It points the data directory at a throwaway temp folder before importing the app, because config.py reads the path at import time. Get the order wrong and your tests would scribble on real data:

Chapter 7
Phase 1 — migrate to SQLite

commit:a6cb1b0

Goal: replace the flat .bin files with a real database — without changing the API, the menu logic, or the frontend at all.

💡 Concept — why a database beats a flat file

A single JSON file means: rewrite the whole file on every change, allow only one writer at a time, and no real way to query ("all spending since March"). SQLite is a complete relational database that lives in one file but gives you transactions (all-or-nothing changes), concurrent reads, and SQL queries — with zero server to run. It's built into Python.

The schema

db.py owns the table definitions. Notice every data table already carries a household_id column — that's groundwork for multi-user support in the next chapter, set to a default of 1 for now:

The connection helper & WAL mode

💡 Concept — transactions & WAL

A transaction groups several changes so they all succeed or all undo together — you never end up half-updated. The get_conn() helper makes that automatic. WAL (Write-Ahead Logging) is a SQLite mode where writes go to a side log first, letting readers keep reading while one writer writes. That's what lifts the old "single process only" limit from Chapter 4's file locks.

The one-time legacy importer

Existing users already have data in .bin files. On first run against a fresh database, db.py imports them — exactly once, guarded by a flag row so it never double-imports even if you later empty everything out:

💡 Concept — idempotency

An operation is idempotent if running it twice does no more than running it once. The legacy_imported flag makes the import idempotent — safe to run on every startup. Idempotency is a recurring theme: schema creation uses CREATE TABLE IF NOT EXISTS, the household seed uses INSERT OR IGNORE.

The seam pays off

Remember the seam from Chapter 2? The migration's whole trick is that grocery.py's public functions keep their exact same names and shapes — only their insides now talk to SQLite instead of a file. api.py, menu.py, and the entire frontend are untouched:

✅ Why it matters

This is the reward for the two earlier investments: the JSON seam (Ch.2) gave a clean swap point, and the characterization tests (Ch.6) proved the swap changed nothing. The only visible difference was that some integers became 1.0 instead of 1 — which fmtQty already renders identically.

Chapter 8
Phase 2 — accounts & households

commit:81e7618

Goal: add logins, and make sure one family's data is invisible to another. The household is the unit of separation; family members are users within a household.

Passwords: hashing, not storing

💡 Concept — hashing vs encryption, and salt

You must never store the actual password. Encryption is reversible (there's a key to get the original back); hashing is a one-way scramble — easy to compute forward, practically impossible to reverse. To check a login you hash what they typed and compare hashes. A salt is random data mixed in so two people with the same password get different hashes (defeating precomputed "rainbow table" attacks). PBKDF2 repeats the hash hundreds of thousands of times to make guessing slow.

The stored string packs everything needed to verify later — algorithm, iteration count, salt, and hash — separated by $. The comparison uses hmac.compare_digest, which takes the same time whether the first or last character differs (more on that in Chapter 9).

Sessions: a token in a cookie

💡 Concept — sessions vs JWT

After login the server needs to remember "this browser is Alice." Cantina creates a random token, stores it in a sessions table, and sends it to the browser as an httponly cookie (JavaScript can't read it, blunting theft via cross-site scripting). On each request the browser returns the cookie; the server looks up the token. This is a server-side session: the token is meaningless on its own and can be revoked instantly by deleting the row — unlike a self-contained JWT, which is harder to revoke.

Multi-tenancy: one switch protects every query

💡 Concept — multi-tenancy & IDOR

Multi-tenancy means many independent customers (here, households) share one app and database, each seeing only their own rows. The classic failure is IDOR (Insecure Direct Object Reference): you ask for "spending entry #5" and the server hands it over without checking it's yours. Cantina's defense is structural — every query filters by the current household, and that household is set in one place per request.

The mechanism is a context variable — a value that's private to each in-flight request. The login dependency sets it; every data query reads it:

Every data route is then placed behind this one dependency, so no individual endpoint needs its own auth code:

⚠️ Pitfall — why get_current_user must be async

This subtlety bit the author. A context variable set inside a synchronous dependency lands in a throwaway worker-thread context and is invisible to the sync endpoint that follows — so queries would silently use the default household. Making get_current_user async means it runs in the request's own task context, so the household value propagates correctly into the endpoint and its database calls. Subtle, and exactly the kind of thing tests for cross-household isolation catch.

Chapter 9
Phase 2.5 — security hardening

commit:12e1382

Goal: a LAN app is one thing; exposing it to the internet is another. This phase closes the gaps a determined attacker would probe, before any remote access is switched on.

✅ Why it matters — harden before you expose

Every measure below was added while the app was still LAN-only. The philosophy: pay the security cost before opening the door, not after the first incident.

Session timeouts

Sessions now have two clocks: an idle timeout (logged out after inactivity) and an absolute cap (logged out regardless), both env-tunable. The idle clock slides forward on each use:

Brute-force throttle & timing-safe login

ratelimit.py keeps an in-memory count of failed logins per (ip, email) with a looser per-IP cap. Too many failures → HTTP 429 with a Retry-After header:

⚠️ Pitfall — email enumeration via timing

If a login for an unknown email returns instantly but a known email takes 50ms (because it actually hashes the password), an attacker can tell which emails are registered just by timing. The fix: when the email is unknown, still verify against a throwaway dummy hash so the work — and the time — is identical:

⚠️ Pitfall — forged proxy headers

Behind a proxy, the real client IP arrives in an X-Forwarded-For header — but that header is attacker-controlled if you trust it blindly, letting one client forge unlimited fake IPs to dodge the throttle. So the code trusts it only when CANTINA_TRUSTED_PROXY=1 is explicitly set (which you do only once a real proxy sits in front).

CSRF protection

💡 Concept — CSRF

Cross-Site Request Forgery: a malicious page you visit secretly makes your browser fire a request to Cantina, riding along on your logged-in cookie. Cantina's defense is a custom header — every state-changing request must carry X-Requested-With: cantina. A cross-site page cannot set a custom header without a CORS grant (which Cantina never gives), so forged requests are rejected. SameSite=Strict on the cookie is a second layer.

💡 Concept — the security headers (CSP & friends)

The middleware also stamps every response with defensive headers: Content-Security-Policy (default-src 'self') tells the browser to only load scripts/images from this site, neutering injected scripts; X-Frame-Options: DENY stops the site being embedded in a hidden iframe (clickjacking); X-Content-Type-Options: nosniff stops the browser guessing file types. HSTS is added only once you're on HTTPS.

Cookies & files at rest

Cookies get hardened: SameSite=Strict, httponly, and — once on HTTPS — the Secure flag plus the browser-enforced __Host- name prefix. On disk, the database holds password hashes and live tokens, so the data directory is locked to owner-only and the DB file is forced to mode 0600:

Finally, the interactive API docs (Swagger) are off by default — they'd otherwise hand an attacker a map of every endpoint. Set CANTINA_ENABLE_DOCS=1 to turn them on for local development.

⚠️ Pitfall — the throttle is single-process

The brute-force counter lives in one process's memory. Run multiple server workers and each keeps its own count, weakening the limit. The code is honest about this in its docstring: a public, multi-process deployment (a future "Phase 4") would move the counter to a shared store like Redis. Knowing a component's limits is part of building it.

Chapter 11
Phase 3 — remote access with a Cloudflare Tunnel

Goal: let the family reach Cantina from outside the house — on a phone, on cellular data — over real HTTPS, without poking a hole in the home router and without making anyone toggle a VPN. The hardening in Chapter 9 was the price of admission; this chapter actually opens the door.

💡 Concept — a tunnel instead of port-forwarding

The obvious way to expose a home server is to forward a port on your router so the internet can connect in. That also means anyone on the internet can connect in — to a machine on your home network. A tunnel inverts this: a small program (cloudflared) running next to the app dials out to Cloudflare's edge and holds the connection open. Public requests arrive at Cloudflare, which pushes them back down that outbound pipe to the app. No inbound ports are opened; your router and firewall stay completely closed.

The only path in is the outbound tunnel.

The cutover: flip the app into "behind TLS" mode

Cloudflare terminates TLS at its edge, so the app itself only ever speaks plain HTTP on loopback. Two things change at cutover, both pre-staged as comments in the systemd unit since Chapter 10:

✅ Why it matters — each flag pays for the new environment

CANTINA_SECURE_COOKIES=1 turns on everything that only makes sense over HTTPS (the Secure cookie, the __Host- prefix, HSTS). CANTINA_TRUSTED_PROXY=1 lets the brute-force throttle from Chapter 9 see the real visitor IP that Cloudflare forwards — without it, every request would look like it came from the tunnel and the per-IP limit would be meaningless. And binding to 127.0.0.1 means even on your LAN there's no longer an open 0.0.0.0 port; the tunnel is the front door.

⚠️ Pitfall — "502 Bad Gateway" means the origin is down

A tunnel can connect to Cloudflare perfectly and still serve 502 Bad Gateway — that error means the edge reached your cloudflared but cloudflared couldn't reach the app. In this build it happened because the cantina.service wasn't running, so nothing was listening on 127.0.0.1:8000. The tunnel being healthy tells you nothing about the origin: when you see 502, check the app, not Cloudflare. (curl -sI http://127.0.0.1:8000/ on the box confirms the origin in one line.)

⚠️ Pitfall — user services die when you log out

Both the app and the tunnel run as user systemd units. By default a user's services stop the moment that user logs out — and don't start at boot until the next interactive login. For an always-on server that silently brings back the 502 after a reboot. The fix is to enable lingering so the user manager runs without a session: loginctl enable-linger <user>. enable-ing the units is necessary but not sufficient without it.

Is the app's own login enough? Defense in depth.

Once the login page is reachable from the whole internet, it's worth asking honestly whether username + password is sufficient. For a closed family group the answer is "a reasonable baseline" — the password storage and session handling from Chapters 8–9 are solid, and crucially there's no self-service signup, so the only public surface is one login form. But two real gaps remain: there's no second factor (a reused, breached password = full access from anywhere), and the brute-force throttle is single-process and in-memory.

✅ Why it matters — add a second, free gate at the edge

The cheapest large win is Cloudflare Access in front of the hostname: an identity check (email one-time code, or Google login) at the edge, before a request ever reaches the login form. Now an attacker needs to pass Cloudflare's gate and know an app password — two independent layers, and the app code didn't change at all. Defense in depth means no single failure (a leaked password, a bug in one check) is game over.

💡 Concept — why the barcode proxy is safe to expose

One endpoint reaches out to the internet on user input: the barcode lookup (Chapter 5). That pattern can be an SSRF risk (tricking the server into fetching an internal address). Here it isn't, for two reasons: the input is validated to be digits only before it's slotted into a fixed URL template, so it can't be twisted into a different host; and the route sits behind the login, so no anonymous visitor can reach it. The one residual concern is a logged-in account using it to hammer OpenFoodFacts through your server's IP — harmless for a family, but a reason a true public launch (a hypothetical Phase 4) would add a per-user rate limit there.

🔧 Try it — verify the cutover from off-network

From a phone on cellular (not your Wi-Fi), open https://your-hostname and check: the page loads and login works with no CSP errors in the console; the session cookie shows __Host-…; Secure; SameSite=Strict (DevTools → Application → Cookies); curl -sI https://your-hostname/docs returns 404; and a few bad logins return 429 with the real client IP in the app log (proving CANTINA_TRUSTED_PROXY is wired through).

⚠️ Pitfall — Cloudflare can rewrite your security headers

The app sends a two-year Strict-Transport-Security header, but the public response showed max-age=0. The origin was correct — Cloudflare's edge was overriding HSTS based on a dashboard toggle (SSL/TLS → Edge Certificates). When a header looks wrong in the browser, compare it against the origin directly (curl -sI http://127.0.0.1:8000/) before suspecting your code: a CDN sits in the middle and is allowed to rewrite.

A build-it-yourself roadmap

If you wanted to build something similar, this order is the lesson:

  1. Model the domain first. Get your core objects and their relationships right before any storage or web code.
  2. Funnel storage through a narrow seam. Two functions (read/write) you can later swap. Use a boring, portable format (JSON) at first.
  3. Add the web layer with validation at the edge. Let a tool like Pydantic reject bad input before it reaches your logic.
  4. Build the UI against that API. Keep it simple; serve it from the same origin to avoid CORS.
  5. Write tests before any risky refactor. Characterization tests turn scary changes into safe ones.
  6. Graduate to a real database only when flat files start to hurt — and let your seam make it a non-event.
  7. Add auth and tenancy structurally, so isolation is enforced in one place, not sprinkled across every endpoint.
  8. Harden before you expose. Threat-model the open internet and close the gaps while you're still safe behind the LAN.
  9. Expose through a tunnel, not an open port, and stack a second gate (an edge identity check) in front of your own login. Reaching the internet is a config step; staying safe there is defense in depth.
✅ The meta-lesson

Almost every good decision here was about making the next change cheap: the JSON seam made the SQLite swap easy; the tests made it provably safe; the household_id columns added in Phase 1 made auth a smaller step in Phase 2. Build so that future-you has an easy time.

Appendix C
Glossary

API / endpoint
A URL the server answers; the set of them is the app's API.
Atomic operation
One that fully happens or not at all — no half-states (e.g. os.replace).
Characterization test
A test that pins current behavior so a refactor can't silently change it.
Context variable
A value private to one in-flight request/task; here, the current household.
CORS
Browser rule restricting calls from one site's page to another site.
CSP
Content-Security-Policy: tells the browser which sources of scripts/images to trust.
CSRF
Cross-Site Request Forgery: tricking your browser into making an authenticated request.
Hashing
One-way scramble; used to store passwords so the original can't be recovered.
HttpOnly cookie
A cookie JavaScript can't read, limiting theft via injected scripts.
Idempotent
Running it twice does no more than running it once.
IDOR
Insecure Direct Object Reference: accessing someone else's record by guessing its id.
JSON
A text format of nested dicts/lists; the lingua franca of web APIs.
Multi-tenancy
Many isolated customers sharing one app/database.
PBKDF2
A deliberately slow, salted password-hashing function.
Pydantic
Library that validates request data against a declared schema.
Salt
Random data mixed into a hash so identical passwords hash differently.
Seam
A narrow boundary you can change behind without disturbing callers.
Serialization
Turning an in-memory object into bytes for storage/transfer.
Session
Server-side record that a given browser is logged in; referenced by a cookie token.
SPA
Single-Page App: one page that JavaScript redraws, fetching only data.
SQLite
A full relational database contained in a single file, built into Python.
Transaction
A group of DB changes that all commit or all roll back together.
uvicorn
The web server that runs a FastAPI app and accepts HTTP connections.
WAL
Write-Ahead Logging: SQLite mode letting readers and one writer work concurrently.
